# ADR-012: Remote Access Transport — WebSocket Gateway with Connection-Decoupled Sessions

**Date:** 2026-05-31
**Status:** Accepted

> **Amendment (2026-06-25, change `gateway-packaging`) — terminology only:** the `Dmon.Gateway` host is renamed `Dmon.Network` (tool command `ndmon`); read "gateway"/"the gateway" as "the network host" throughout. No numbered decision changes — see ADR-033.

## Context

`dmoncore` speaks JSONL over stdio (ADR-003): the host writes commands `{"id","type",…}` to the core's stdin and reads events `{"id","event",…}` from its stdout. This is already a full-duplex, asynchronous, id-correlated message stream — stdin is an ordered, flow-controlled command channel; stdout is the event channel. Today the only consumer is a local host (`Dmon.Terminal`) that spawns the core as a child process via `Dmon.Runtime`'s `CoreProcessManager` (ADR-011).

A new requirement is to expose the core **across a network boundary**. The first consumer is an **iOS application written in Swift**, operating over **mobile cellular (4G/5G) and WiFi**. The server side must hold a **long-lived handler** — the thing that owns the `dmoncore` process for a session — and that handler must **outlive any single connection**, so a reconnect carrying a session id immediately re-attaches to the running core. The gateway runs on the user's **home server**, reached over **Tailscale** (chosen over a Cloudflare Tunnel for **zero public exposure** — nothing is published to the public internet; see Decision 12) and is **permanently single-tenant** (exactly one principal, not a V1 simplification). That topology shapes the auth model (Decisions 10 and 12): Tailscale, not the gateway, is the authentication and encryption boundary.

This is squarely the **"Remote agent execution"** item the V1 brief defers. This ADR is the vehicle for bringing it into scope; acceptance requires `coding-agent-brief.md` to be updated to match. It is **Proposed**, not Accepted.

Two properties of the deployment shape the transport choice and dominate everything else:

- **iOS-native client support.** WebSocket has first-class, dependency-free Swift support (`URLSessionWebSocketTask`, Foundation, iOS 13+). gRPC needs `grpc-swift` + a proto schema maintained alongside the JSON wire shape; SignalR's Swift client is young; SSE has no Foundation client.
- **Cellular reality.** Carrier NAT silently drops idle TCP flows; WiFi↔cellular handoff changes the device IP and *drops* the connection; iOS **tears down every socket when the app backgrounds**. No transport keeps a socket alive across these. The connection is therefore expected to drop routinely, and **the session must be decoupled from the connection** — the physical transport matters less than the resumption layer above it.

The protocol is already a full-duplex ordered stream. The cheapest transport is the one that preserves that with the least translation, and whose upstream (command) direction keeps wire-order and TCP flow control for free — rather than shattering it into independently-acknowledged, reorderable requests (the upstream-backpressure cost of an SSE-down/POST-up model).

## Decision

1. **Transport is a raw WebSocket.** A new ASP.NET Core host exposes a WebSocket endpoint (`app.UseWebSockets()`, Kestrel). One full-duplex channel per attached connection; ADR-003 command/event JSON is forwarded to and from the core's stdio **near-verbatim** as text frames. This inherits TCP flow control and wire-order on the command direction (matching the stdio pipe the core already speaks) and is the only candidate with first-class, dependency-free Swift client support. Transport encryption is provided by Tailscale (Decision 12): `tailscale serve` fronts the loopback-bound gateway and terminates TLS with a Let's Encrypt certificate for the server's MagicDNS name, so the iOS client gets valid `wss://` without the gateway managing certificates.

2. **Sessions outlive connections — a new `Dmon.Gateway` host.** A **`SessionHandler`** owns exactly one `dmon-core` process for one `sessionId`, spawned through `Dmon.Runtime`'s `CoreProcessManager` (ADR-011 — `Dmon.Runtime` carries no console dependency, so the gateway inherits the same core bootstrap the terminal and a future desktop host use). Handlers live in an in-memory **session registry** (`ConcurrentDictionary<sessionId, SessionHandler>`). A WebSocket **attaches** on connect and **detaches** on drop; the handler persists across detach. The stdio pump binds to the *handler*, not the connection: client frames → core stdin; core stdout → the currently-attached connection, or to durable buffering when none is attached.

3. **The gateway adds a connection-control sub-protocol; ADR-003 is unchanged.** Sequencing, acknowledgement, resume, and heartbeat are gateway concerns layered *around* the unchanged ADR-003 messages on the WebSocket — not new core stdio messages. The core protocol stays intact; the resilience protocol is localized to the network boundary. The control frames are: `attach` (client→gateway: `sessionId`, `lastSeq` — the bearer token rides the upgrade request, **not** this frame, per Decision 12), `attached` (gateway→client: `generation`, `headSeq`), `ack` (gateway→client: command `id` received), `ping`/`pong` (both directions).

4. **Event sequencing and replay-on-reattach.** The gateway assigns a **monotonic per-session `seq`** to every server→client event as it flows out, durably backed by the session's append-only `messages.jsonl` (ADR-004). On reattach the client sends its `lastSeq`; the handler **subscribes to the live stream first** (into a buffer), then **replays `(lastSeq, headSeq]`** from the log, then flushes the buffer deduping by `seq` — closing the catch-up seam without dropping or duplicating events that arrive mid-replay.

5. **Command idempotency across the gap.** Commands carry the `id` ADR-003 already defines. The handler **acks each command by `id`**; on reattach the client **resends any unacked commands**; the handler **dedups by `id`**. This prevents a drop between send and ack from double-submitting a turn.

6. **Stale-connection fencing and single active writer.** Each attach issues a monotonically increasing **generation token**; the handler drops frames from any connection older than the current generation and closes that socket. A new attach to a live handler **evicts the prior connection** (fences it, closes it) — a single active writer per session. This is the correct default for "aggressive reconnect," where the *old* socket is frequently not yet known-dead (carrier NAT merely stopped delivering) when the new one arrives. Read-only multi-observer attachment (e.g. iPad + iPhone watching one session) is **deferred** (Open Question A).

7. **Detached-handler lifetime is running-turn-aware.** On *detected* detach the handler enters a detached state and starts a grace timer:
   - **Idle-detached** (no turn in flight) → reaped after a short configurable TTL.
   - **Detached with a turn in flight** → kept alive until the turn completes (killing it wastes paid LLM work), after which the idle TTL applies; bounded by an absolute maximum.
   The TTL floor must accommodate mobile backgrounding + handoff gaps (think minutes, not seconds). A hard cap on concurrent live handlers is the resource ceiling. All three values are config.

8. **Liveness via application-level heartbeat.** `ping`/`pong` on a ~20–30s interval beats carrier NAT idle-timeouts and detects dead/ghost sockets promptly. The detached-TTL clock (Decision 7) starts on *detected* disconnect, so detection latency is part of the policy.

9. **Single server instance for V1.** `sessionId → SessionHandler` is **in-process affinity**: a live OS process cannot migrate between machines, so a reconnect must reach the instance holding the core. V1 is one gateway process, in-memory registry, no backplane. Multi-instance sticky routing (consistent hashing / session-aware LB) is a **future ADR** — this is precisely the scale-out problem a SignalR backplane would solve, which V1 deliberately does not need.

10. **Client→gateway auth is a distinct axis from ADR-005, but delegated to the deployment.** ADR-005 governs **gateway→provider** credentials (API keys, server-side); the **client→gateway** axis is a separate concern. Because the gateway is single-tenant behind a private overlay (Context), that axis is handled by the **trusted network boundary plus an optional shared key** (Decision 12), not by per-session bearer tokens. `sessionId` remains a **routing/capability key, never an auth credential** (or it is a session-hijack vector); with exactly one principal, ownership verification collapses to "the client reached the trusted interface."

11. **The permission model holds across the remote edge.** ADR-006's prompts (`tool.confirmRequest`, `ui.inputRequest`) flow to the remote client over the same channel. While a session is **detached**, a turn that reaches a permission gate **parks** — it must not auto-approve (violates ADR-006's conservative default) nor auto-deny (silently kills work) — and resumes when the client reattaches, or times out under the handler's TTL (Decision 7). Conservative-by-default is preserved over the network.

12. **Client→gateway auth: Tailscale is the boundary; an optional shared key adds defense-in-depth.** (Concretizes Decision 10; resolves Open Question B.) The gateway runs on a home server reachable only over **Tailscale** and is permanently single-tenant, so the elaborate per-device token model an open-internet edge would need is **deliberately not built** — Tailscale already provides device identity, end-to-end encryption, and per-device revocation.
    - **Deployment shape.** The gateway binds to **loopback**; **`tailscale serve`** fronts it on the tailnet and terminates TLS with a Let's Encrypt certificate provisioned for the server's MagicDNS name. The iOS client connects to `wss://<host>.<tailnet>.ts.net`. The gateway never listens on a public NIC.
    - **The network is the primary control.** Device identity, per-device revocation, and user authentication are delegated to Tailscale (tailnet membership, device management, ACLs) — dmon does not reimplement them. A tailnet ACL restricts the iOS device to the gateway's port (least privilege).
    - **Single tenancy collapses ownership.** With exactly one principal, Decision 10's session-ownership check reduces to "the client reached the gateway over the tailnet." No principal/device matrix, no token store, no pairing enrollment — and a session started on one device resumes on another for free (covering Open Question A's cross-device case).
    - **Optional shared key.** The gateway may additionally require a single pre-shared key — generated at first run, stored once in the iOS **Keychain**, sent as `Authorization: Bearer <key>` on the WebSocket upgrade; a mismatch **rejects the upgrade with HTTP 401**. One config value, one comparison — defense-in-depth against a misconfigured tailnet ACL or a compromised tailnet device.
    - **Fencing is unchanged.** Decision 6's generation/evict-old fencing is connection hygiene for the reconnect storm, independent of auth, and stays as-is.
    - **iOS connectivity is the client's choice, not the gateway's concern.** V1 expects the **system Tailscale app** (mature; the tailnet path persists across app backgrounding, aiding reconnect). Embedding `libtailscale` / TailscaleKit for a self-contained app (no dependency on the Tailscale app) is a future option and changes nothing server-side.

## Consequences

- **The core is reachable over the network with minimal protocol translation.** Because the gateway forwards ADR-003 JSON near-verbatim and confines resilience to a control sub-protocol, the core and the terminal host are untouched; the wire contract does not change.
- **The connection is disposable; the session is durable.** Routine cellular drops — backgrounding, handoff, NAT timeout — become reconnect-and-replay rather than lost work, leaning on `messages.jsonl` (ADR-004) as the resume cursor.
- **A backgrounded app cannot be *pushed* a result by any transport.** Delivering "your turn finished" to a suspended iOS app needs **APNs** (out of this ADR's transport scope but a hard product dependency), paired with resume-on-foreground via `lastSeq`. This couples directly to the detached-TTL policy (Decision 7): how long a detached handler is retained is how long a backgrounded user has to come back.
- **Server-crash loses the in-flight turn but not the session.** All handlers and their cores die with the gateway process; reconnect replays history from disk and starts a fresh handler, but an interrupted turn is gone. This rhymes with the existing `/reload`-loses-history gap and should be decided alongside any core turn-persistence work, not twice.
- **New host project and an ASP.NET Core dependency.** `Dmon.Gateway` is a third host surface beside `Dmon.Terminal`; it depends on `Dmon.Runtime` (ADR-011) for core bootstrap and on ASP.NET Core for the WebSocket endpoint. No new dependency reaches the core or the contract packages.
- **Scope expands.** Accepting this ADR brings "Remote agent execution" into scope; `coding-agent-brief.md` and `CLAUDE.md`'s out-of-scope list must be updated as part of acceptance.

## Alternatives

- **gRPC bidirectional streaming.** Works natively on iOS (real HTTP/2, no gRPC-Web needed) and gives typed contracts and HTTP/2 flow control. Rejected for V1: the bidi stream dies on every handoff/background so reconnection is manual *anyway*, and it forces a protobuf schema maintained beside the JSON wire shape — paying the structure tax without the structure addressing the actual hard problem (resumption). The JSON would otherwise just ride in a `bytes` field, defeating the typing.
- **Streamable HTTP (SSE down + POST up), the MCP transport shape.** Best carrier/proxy traversal and resumption-by-design (`Last-Event-ID`), and MCP-client compatibility. Rejected for V1 because the upstream (command) direction is not a stream: independent POSTs lose wire-order and TCP flow control, forcing app-level sequencing, reordering, and a bounded queue with `429`/backoff — backpressure machinery WebSocket gets for free. Reconsider if MCP-client compatibility or hostile-proxy traversal later outranks a single full-duplex channel.
- **SignalR.** Managed auto-reconnect, stateful-reconnect buffering for brief blips, and transport fallback (WebSocket→SSE→long-poll) genuinely useful on weird carrier networks; backplane for future scale-out. Rejected for V1: its reconnect does not survive backgrounding-length gaps without the same app-level resume we build anyway, the Swift client is immature, and the hub framing wraps our messages. Revisit if scale-out (Decision 9) and auto-fallback become requirements.
- **Raw TCP/TLS via Kestrel `ConnectionHandler`.** Lowest translation — point the byte protocol at a `NetworkStream`. Rejected: carrier-NAT/proxy-hostile, no resume semantics, no browser/standard tooling, and it rebuilds auth/routing/observability HTTP provides.

## Open Questions

- **A. Multi-observer attach.** V1 is single active writer with evict-old (Decision 6). Do we later allow additional **read-only** connections (same session on a second device) observing the event stream without writing commands?
- **B. Token issuance/format.** *Resolved — see Decision 12*: **Tailscale** is the auth + encryption boundary (`tailscale serve` + Let's Encrypt cert), with an optional single shared key for defense-in-depth. Single-tenant and permanent; per-device identity and revocation are delegated to Tailscale (tailnet membership + ACLs).
- **C. Session creation vs resumption.** *Partially resolved by ADR-013.* This ADR specifies *re-attach* by `sessionId`. The first-contact flow — how a client creates a brand-new session and obtains its `sessionId` — still needs its own `createSession` control message (shape unspecified here). What that message *carries* is now settled: a **`profile`** (ADR-013), which determines the persona, whether an `assets/<session_id>/` directory is provisioned, and the permission mode. Session creation therefore allocates the `sessionId`, provisions the per-session storage dir (ADR-004) and — iff `profile.assets` — the durable asset dir, then spawns the handler.
- **D. Replay bound / compaction.** A client absent for a long turn may face thousands of events on replay. V1 replays the full `(lastSeq, headSeq]` range; whether a snapshot/compaction story is needed is deferred.

## Relationship to other ADRs

- **ADR-003** — the binding host↔core contract is unchanged. The gateway forwards command/event JSON verbatim and adds a *connection-control* sub-protocol (attach/ack/resume/heartbeat) only on the WebSocket boundary; no core stdio message shapes change.
- **ADR-004** — the session directory and append-only `messages.jsonl` are the durable backing for event sequencing and replay (Decisions 4, 7); the gateway adds a per-session `seq`, sourced from or aligned to the log.
- **ADR-005** — provider auth (gateway→provider, API keys) is untouched. The orthogonal **client→gateway** axis (Decisions 10, 12) is delegated to Tailscale (tailnet membership + ACLs) plus an optional shared key, not to per-session tokens; the provider credential model is unaffected.
- **ADR-006** — the conservative permission model is preserved over the network: prompts flow to the remote client, and a detached session **parks** at a permission gate rather than auto-resolving (Decision 11).
- **ADR-011** — the gateway reuses `Dmon.Runtime`'s `CoreProcessManager` for core discovery/spawn/lifecycle; `Dmon.Runtime`'s console-free design (ADR-011 Decision 7) is what lets a third host surface inherit identical bootstrap. The core is acquired and run exactly as for the terminal host.
- **ADR-013** — `createSession` carries an agent **profile**, which decides the persona, the per-session asset directory (default off), and the permission mode. The asset directory is durable and outlives the handler reaped under Decision 7; this resolves Open Question C's payload.
