# DEVLOG — dmonium-windowed-dashboard

> **Summary (pinned):** Converting `dmonium` (`daemon/Daemon.App`, a Swift Package) from menu-bar-only into a window-primary dashboard with an optional default-off menu-bar icon. Architecture is design.md D1–D7; the lifecycle crux is D1/D2 (a single `DaemonController` owner bootstrapped once from `applicationDidFinishLaunching`, never a view `.task`). Gates are **Swift, not .NET**: `swift build -c release --package-path daemon/Daemon.App`, `swift test --package-path daemon/Daemon.App`, `openspec validate dmonium-windowed-dashboard --strict`. Branch `change/dmonium-windowed-dashboard`.

## Group 1 — DaemonController owner + window-independent bootstrap

### Block 1.1–1.2 — `ComponentHealth.lastUpdated` + stamping (DONE)

- **What:** Added `lastUpdated: Date?` (default `nil`) to `ComponentHealth` and stamped it at all six publish sites. Data-model + stamping half of **D6**; the *rendering* ("12s ago" in the dashboard grid) is deferred to task **3.3**.
- **Two publish shapes (worker/architect note):** publishers construct `ComponentHealth` two ways — (a) declarative Combine `.map` in `init` (`GatewayManager`, `ServiceManager`, `TailscaleMonitor`) where `Date()` must be stamped *inside* the closure so it evaluates per-emission, and (b) imperative assignment in poll/result handlers (`DcalHealthMonitor`, `DmailHealthMonitor`, `EndpointHealthProbe`). Both handled.
- **Seed values stay `nil`:** the initial `.unknown`/`.down` seeds are deliberately left unstamped — `lastUpdated == nil` is the "never published yet" semantic. This also keeps the existing rollup/classifier tests valid (the `component(_:)` test helper omits `lastUpdated` → `nil` on both sides of equality).
- **Rollup untouched:** `lastUpdated` does NOT enter `HealthRegistry.rollup` (still `name/status`-only); `HealthRegistry.swift` unchanged.
- **Tests:** +3 deterministic tests in `HealthClassificationTests.swift` (seed isNil; stamped isNonNil; two stamps non-decreasing via `<=`, no flake). Suite 31 → 34, all green.
- **Gates:** release build clean (0 warnings); `swift test` 34/0; `openspec validate --strict` valid. Reviewer: **Approve**, no blockers/nits.
- **Carry-forward:** spec "Unified health surface" also requires the dashboard to *render* the timestamp — closed later by **3.3** (Group 3). Reviewer flagged this as correctly out-of-scope here.

### Block 1.3–1.5 + 2.1–2.3 + 5.1 — `DaemonController` lifecycle owner (DONE)

- **What:** Introduced `DaemonController` (`@MainActor final class … : ObservableObject`) owning all 11 managers/monitors/probes + `HealthRegistry`. `bootstrap()` (guarded by `private(set) var hasBootstrapped`, flag set *before* side-effects) starts every manager, registers the nine health publishers (orders 0–8) verbatim, wires `observeGatewayStopped`. `shutdown()` = `gateway/dcal/dmail .stop()` (exact prior teardown). `DaemonApp` collapses its ~11 `@StateObject`s to one `@StateObject private var controller`; the `.task { }` startup block is **removed entirely**. `AppDelegate` gains `applicationDidFinishLaunching` (`.regular` activation + `controller.bootstrap()`), `applicationShouldTerminateAfterLastWindowClosed → false`, and `applicationWillTerminate → controller.shutdown()`.
- **Single-instance wiring (the key decision):** `AppDelegate` owns `let controller = DaemonController()`; `DaemonApp.init()` seeds its `@StateObject` from the already-constructed adapted delegate via `_controller = StateObject(wrappedValue: existingDelegate.controller)` (hoisting `let existingDelegate = _appDelegate.wrappedValue` to dodge the escaping-autoclosure-captures-self diagnostic). One instance: `bootstrap()` fires on the same object the `.environmentObject(...)` chain observes. **Reviewer note (important if reused):** this is safe *only* because `App` is a process-singleton (the `StateObject(wrappedValue:)` autoclosure runs once) — the same pattern in a `View` (init runs many times) would be a latent bug.
- **Scene unchanged:** kept `MenuBarExtra(isInserted: $isInserted)` (`@State … = true`) + `Settings` scenes, rebound to `controller`. **No `WindowGroup` yet** (Group 3).
- **1.4:** confirm-only — all four monitors already had `guard pollTask == nil`; `ServerProcessManager` adoption-guarded. Zero code change.
- **Tests:** +4 `@MainActor` idempotence tests (`DaemonControllerTests.swift`); 34 → 38, all green.
- **Gates:** release build clean (0 warn); `swift test` 38/0; `openspec validate --strict` valid. Reviewer: **Approve with nits** (no blockers).
- **NOTE — SourceKit false positives:** the IDE reported "Cannot find … in scope" / "No such module XCTest" after this block; these are stale-index artifacts (SwiftPM module, no Xcode project). Real `swift build -c release` / `swift test` are green — verified directly. Don't chase these.
- **CARRY-FORWARD for the Group 3 block (reviewer nit 2):** `HealthRegistry.swift` doc comments still say "Set by the wiring in DaemonApp.task" / "Call once per component in DaemonApp.task" (lines ~36-37, 47, 63-64, 72) — the `.task` is gone; wiring now lives in `DaemonController.bootstrap()`. Left untouched (out of this block's scope). Fix when Group 3 touches `HealthRegistry` for timestamp rendering (3.3).
- **Reviewer nit 1 (non-blocking):** test 5.1 proves "second `bootstrap()` is a no-op" via the `hasBootstrapped` guard, but doesn't directly assert "exactly nine sinks, a dup would leak" (`HealthRegistry.cancellables` is private, no test seam). Acceptable; a future block could expose a count if a sharper assertion is wanted.

## Group 3 — WindowGroup dashboard

### Block 3.1–3.5 — `WindowGroup` dashboard (DONE, human-visual confirmed)

- **What:** Primary scene is now `WindowGroup(id: "dashboard") { DashboardView().environmentObject(controller) }`; the separate `Settings` scene is **removed**. New `DashboardView.swift` = `NavigationSplitView` with a `CaseIterable enum DashboardSection { status, services, settings }` sidebar; `StatusSectionView` (health grid: status icon + name + detail + relative last-updated time), `ServicesSectionView` (Gateway/Dcal/Dmail start/stop/restart + Sync-Calendar + Bring-Tailscale-up), reuses `SettingsView` verbatim as the `.settings` pane (env-injected `controller.gateway`). New `StatusHelpers.swift` hoists `statusSymbol`/`statusColor` (shared with `MenuBarView`) + pure `relativeAge(from:now:)` (`nil → "—"`). `MenuBarExtra` kept present-and-on (`isInserted = true`) — the default-off toggle is still **4.2**.
- **B1 fix (reviewer-found blocker):** removing the `Settings` scene killed the `showPreferencesWindow:` responder, so `MenuBarView`'s "Open Settings…" (`NSApp.sendAction`) became a dead no-op. **Fix:** promoted section selection to a single source of truth — `@Published var selectedSection: DashboardSection` on `DaemonController`. All three entry points converge on it: sidebar `List(selection:)` binds to it; the ⌘, `CommandGroup(replacing: .appSettings)` button sets `.settings` + `NSApp.activate`; the menu-bar button sets `.settings` + `openWindow(id: "dashboard")` (reopens/focuses since close≠quit can leave the window closed) + `NSApp.activate`. Dead `sendAction` removed.
- **N1 fix:** Status `ForEach` keyed by `\.name` (was `\.offset`). N2 (scope the free funcs) skipped — optional.
- **Doc carry-forward DONE:** the stale `DaemonApp.task` comments in `HealthRegistry.swift` corrected (now reference `DaemonController.bootstrap()` / "status grid").
- **Tests:** +5 deterministic `RelativeAgeTests` (`nil→"—"`, 12s→"second", 2h→"hour", same-instant non-placeholder, injected-`now` determinism). 38 → 43, all green. View layout itself is not headlessly testable.
- **Gates:** release build clean (0 warn; SourceKit "cannot find in scope" / "SelectionValue could not be inferred" / "No such module XCTest" all confirmed STALE-INDEX FALSE POSITIVES via a forced clean recompile); `swift test` 43/0; `openspec validate --strict` valid. Reviewer (2 rounds): **Approve / sign-off code-clean pending human-visual.**
- **HUMAN-VISUAL GATE SATISFIED (user, 2026-06-23):** user launched the built executable and confirmed window renders, sidebar nav, ⌘,-selects-Settings-in-window, menu-bar "Open Settings…" reopen/focus, close≠quit, and live row redraw all work — *"apart from the Gateway, everything seems to work."*

So the Gateway row sits red out-of-the-box (start() no-ops when no candidate resolves). NOT caused by this change; do NOT fix it here (scope). Follow-up: a `make gateway` target + likely `PackAsTool`/`ToolCommandName`, or document `DMON_GATEWAY_PATH`.
