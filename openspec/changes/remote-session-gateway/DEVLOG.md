## Decisions & deviations

- **§4 — ADR-012 Decision 4 was factually unimplementable; resolved by new ADR-014.** Implementing event sequencing revealed Decision 4's premise ("seq durably backed by `messages.jsonl`") is false: the core's ADR-003 event stream is emitted to stdout by `EventEmitter` and **never persisted**; `messages.jsonl` is written only by the core (`ShortTermMemory` conversational turns + `SessionStore`), and `IMessageAppender` is single-writer-per-session. A separate gateway process can't use it as the event-replay backing without corrupting the core's log / racing its writer. **Stopped and asked the user** (the spec/ADR was wrong — CLAUDE.md §4). User chose **in-memory seq buffer + ADR amendment** (over a gateway-owned durable event log, and over the non-viable write-into-`messages.jsonl`). Wrote **ADR-014** (Amends ADR-012 Decision 4): in-session reconnect replay `(lastSeq, headSeq]` is served from the live handler's in-memory seq-indexed buffer (the handler outlives the connection); cross-restart recovery is the core loading `messages.jsonl` conversational history, with the interrupted in-flight turn lost (already an accepted ADR-012 risk). Amended `spec.md` / `design.md` D2 / `proposal.md` / `tasks.md` 4.1-4.2 / `CLAUDE.md` ADR table to match; got user OK before implementing; folded ADR-014 + amendments into commit `82c0c17`. Memory: `[[adr-014-gateway-replay-not-messages-jsonl]]`.
- **§2 — single ordered drain loop chosen over fire-and-forget flush.** Reviewer's first-pass blocker showed the original fire-and-forget buffer flush on attach would let live events overtake buffered events across the reconnect seam. Reworked to one reader task + one drain loop (the sole sender). This single ordered loop is what §4's replay and §6's fencing both build on — a load-bearing architectural choice made under review pressure.
- **§3 — single serialized send funnel.** Reviewer caught that the §3 endpoint added a second writer (`attached`/`pong`) to the same socket as §2's drain loop → illegal concurrent `WebSocket.SendAsync`. All gateway→client bytes now funnel through one `SemaphoreSlim`-serialized `WebSocketGatewayConnection.SendAsync`. §6's eviction and §7's heartbeat must use this same funnel, not the raw socket.
- **§5 — H1, false-ack-on-failure fixed rather than accepted-as-risk.** Record-before-write is correct for the concurrent-duplicate case but, on a core-write failure, would silently drop the command AND send a false re-ack on resend — defeating ADR-012 Decision 5. Chose the higher-bar fix (compensate admission on failure + ack only after a successful write) over recording it as an accepted V1 risk.
- **§5/§6 nits — input hardening + explicit fencing flag.** Hardened both control-frame parsers (`GetCommandId`/`GetGwDiscriminator`) to read top-level fields type-tolerantly (untrusted edge); replaced §6's magic-`0` "fencing off" with an explicit `enforceFencing` flag (production always enforces).
- **§7 — turn-in-flight signal is `turnStart`/`turnEnd`, NOT `agentStart`/`agentEnd`.** The brief (echoing the loose "agentStart/turn boundaries" wording in design D4 / the old resume note) initially told the worker to key in-flight on `agentStart`/`agentEnd`. The reviewer verified the core **never emits** those — `AgentStartEvent`/`AgentEndEvent` are dead taxonomy entries (no `new …` anywhere; only a no-op `ConsoleEventHandler` arm). The real boundary is `TurnHandler.RunTurnAsync` emitting one `turnStart` (`:257`) → running the **entire** agentic loop (all LLM iterations + tool calls + follow-ups) → one `turnEnd` (`:395`, on the normal *and* cancellation paths). So `turnStart`…`turnEnd` brackets the whole "agent working" window with no inner gaps — the correct in-flight signal. **Follow-up cleanup (separate change):** `agentStart`/`agentEnd` are misleading dead protocol surface that invited this bug; consider removing them from the ADR-003 taxonomy.
- **§7 — `error` must not clear in-flight.** `error` events fire mid-turn in many non-terminal contexts (`turnInProgress` while a turn is running, unknown-confirm/ui-input ids, parse faults). Clearing in-flight on `error` would let a still-running turn's handler reap at the idle TTL. In-flight is bracketed solely by `turnStart`/`turnEnd`; a turn that ends without a `turnEnd` (e.g. core crash) is covered by the `RunningTurnTtlMinutes` absolute-max bound, not by mis-clearing on `error`.
- **§7 — heartbeat made deterministically testable.** First-pass test used a real wall-clock 1s interval + multi-second wait (the worker found `FakeTimeProvider` "unreliable") and asserted only `loopTask.IsCompleted` (any completion cause). Reworked so the production heartbeat drives both the interval and the missed-beat deadline via the injected `TimeProvider`, and the test advances `FakeTimeProvider` and asserts the detach was caused by a *missed beat* (a `ping` was sent AND the loop exited via `heartbeatCts`, with no client-side cancellation/close). The two ~50 ms yields remaining are scheduling barriers (let the heartbeat continuation register its next `Task.Delay(TimeProvider)` before the clock advances), not timing assertions.
- **§7 — removed dead `DetachedTtlMinutes` option (N1).** A fourth TTL knob added speculatively in §1, redundant with `IdleDetachedTtlMinutes` under D4's three-value model (idle TTL / absolute max / concurrent cap) and read by nothing. A silently-ignored knob in a lifetime/security config is a footgun; removed from `GatewayOptions` + `appsettings.json`.
- **§8 — ROBUST parking scope chosen (user decision), not the spec-literal minimum.** The two spec scenarios ("gate reached *while detached*" → held; "reattach while parked" → delivered) are already satisfied for free by §4 replay + §7 keeping the in-flight handler alive — they need zero new code. **Stopped and asked the user** because the requirement text ("On reattach the parked request SHALL be delivered") under the mobile failure modes that motivate ADR-012 implies more than the literal scenarios: the common case is the client *saw* the prompt, then iOS backgrounded/killed the app before the user answered — and plain replay won't resurface a prompt the client already advanced past (`seq` is gateway-internal, not in the payload; the client tracks position from the attach `headSeq` baseline). User chose **robust re-surface** over spec-literal. Implemented: per-handler outstanding-request tracking + reattach re-emit of requests with `OriginalSeq ≤ cursor`.
- **§9 — wildcard bind is refused even under opt-in.** "Loopback by default + explicit opt-in to any other interface" (design Risk row) plus "never bind a public NIC" (spec) map to: loopback always OK; a *specific* non-loopback host (e.g. a Tailscale `100.x` or LAN IP) requires `AllowNonLoopbackBind=true`; a *wildcard* (`0.0.0.0`/`::`/`*`/`+`) binds all interfaces incl. any public NIC and is therefore **hard-refused regardless of the opt-in flag**. Empty/unparseable fails closed. The opt-in means "I have a specific interface in mind," categorically distinct from "bind everything." Worker agreed; reviewer confirmed wildcard is checked before the loopback/opt-in branches. Tailscale (`tailscale serve` fronting loopback) remains the intended exposure path, not a direct non-loopback bind.
- **§9 — 401 has no `WWW-Authenticate` header (deliberate).** For a Tailscale-fronted, non-interactive Bearer scheme, omitting `WWW-Authenticate` avoids advertising the auth mechanism to a scanner. Plain 401 satisfies the spec. Reviewer noted; flag for the §11.4 deployment docs.
- **§8 — re-emit at a FRESH seq, not the original, and never out-of-band.** A prompt the client advanced past can't be re-delivered by lowering the single drain cursor (would replay everything in between, breaking §4 exactly-once). Re-sending the original `seq` out-of-band would desync the client's position. So an outstanding request the replay window won't cover is **re-emitted as a new `_seqLog` entry above `headSeq`** and delivered by the normal drain — internally consistent with §4 regardless of how the (out-of-scope) client tracks position. Load-bearing assumption: the client keys permission prompts by request **`id`** (idempotent), which the core's own dedupe backs (`tool.confirmResponse`/`ui.inputResponse` match `cmd.Id`; a late/duplicate response gets a recoverable `unknownConfirmId`/`unknownUiInputId`). **This client-side idempotency contract must be carried into the iOS-client spec.** Bounded growth: one re-emit per reconnect-while-unanswered (same V1 no-truncation tradeoff as §4/ADR-014).

## Open follow-ups / known gaps (after this change lands — NOT in scope here)

- **Event-stream durability across a gateway restart:** explicitly NOT provided (ADR-014). An interrupted in-flight turn is lost on restart; conversational continuity comes from the core loading `messages.jsonl`. Couples to any future core turn-persistence work. Memory: `[[adr-014-gateway-replay-not-messages-jsonl]]`, `[[followup-turn-persistence-across-restart]]`, `[[shortterm-memory-jsonl-double-write]]`.
- **Replay bound / compaction (ADR-012 Open Question D):** the in-memory seq buffer is unbounded for the life of the handler (V1 accepted, design Risks). The retention ceiling + truncation-signal protocol is deferred. Memory: `[[adr-014-gateway-replay-not-messages-jsonl]]`.
- **Core-write-failure close reuses `ProtocolViolationFrameException` (status 4500):** acceptable for §5, flagged to revisit when real core-death/reaping handling lands in §7. §7 added the reaper but did not change this close path — still acceptable.
- **Reap-vs-same-handler-reattach micro-race (§7, reviewer architectural note):** `Remove`-before-`StopAsync` closes the reap-vs-*new-lookup* race (a fresh `attach` after `Remove` sees nothing → 4404). It does **not** close a same-handler reattach landing in the microsecond window between the reaper reading `DetachedAt`/`IsTurnInFlight` and the `Remove` (`Attach` mutates the handler in place and never re-registers; the reaper does not re-check after `Remove`). Consequence: a just-reattached client is torn down and must reconnect — the same recoverable outcome the resumable-session design already tolerates; scan cadence is 30s; single-tenant home server. Not blocking; worth a one-line `design.md` note or a follow-up if it ever bites.
- **`agentStart`/`agentEnd` dead protocol surface:** in the ADR-003 event taxonomy with no emitter and only a no-op consumer; invited the §7 B1 bug. Candidate for removal in a separate change.
- **Client-side permission-prompt idempotency (§8):** the robust re-surface relies on the iOS client keying prompts by request `id` and tolerating a prompt re-appearing under a new `seq`. This contract must be made explicit in the client spec / Group covering the iOS app (reviewer architectural note).
