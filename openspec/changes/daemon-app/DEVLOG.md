## Section 11 — Daemon.Routing test suite (DONE)

- **11.1–11.5 complete; reviewer APPROVED after two fix rounds.** New `test/Daemon.Routing.Tests/` (22 tests): routing invariants (incl. egress-first precedence + threshold boundary), privacy invariant both directions (distinct `personal_tool`/`world_tool` names), exactly-once counter via `MeterListener`, streaming ack, the terminal-client hook (drives public verbs → `ITerminalClientFactory.Create` → `IsType<TriageRouter>`), and the A1 fail-safe. Added to root `daemon.slnx` + `Everything.slnx`. Gates green (`make build` 0/0; full `make test` verified green twice by the orchestrator; `validate --strict`).
- **CARRIES A §4 PRODUCTION FIX (the tests caught a real bug the §4 review missed):** `TriageRouter`'s A1 fail-safe used `classifyResponse.Result ?? default`, but M.E.AI 10.5.2 `ChatResponse<T>.Result` **throws `JsonException`** on unparseable JSON (it never returns null) — so the fallback was dead code and a bad classification crashed the turn. Fixed to `TryGetResult(out RouteDecision? parsed) && parsed is not null ? parsed : new RouteDecision("personal", Tier.Direct, false, 1f)` (non-throwing). **Lesson:** prefer `TryGetResult` over `Result` for structured output.
- **Spec/design realignment (orchestrator):** the §11 tests also exposed that design.md R5 prose said reasoner dispatches "regardless of scope", contradicting the `triage-routing` spec's egress-first order. The SPEC is authoritative and self-consistent (egress when world+impersonal+confident; **otherwise** reasoner when Tier.Reasoner). Production code was already correct (egress-first); design R5 prose corrected to match; the mis-encoded test was rewritten (`EgressConditionMet_ReasonerTier_RoutesToEgress_NotReasoner`).
- **Flaky-test fix:** `DaemonTelemetry`'s counter is a process-global `Counter<long>`, so a `MeterListener` sees measurements from ALL parallel test classes → counter delta bled to 2 under xUnit's default cross-class parallelism. Fixed with `[assembly: CollectionBehavior(DisableTestParallelization = true)]` in the test assembly; all 5 counter-measuring tests route through one `MeasureCounterDeltaAsync` helper. **Arch note (for whoever extends the telemetry surface):** if counter-correctness tests proliferate, add an injectable recorder / test `IMeterFactory` seam so tests observe an isolated meter instead of the process-global one.
- **Pre-existing flake (NOT introduced here, noted for CI):** the worker hit a one-off `Dmon.Core.Tests` failure in a full `make test` run (591/591 in isolation); it did NOT reproduce in two orchestrator full-suite runs. Latent cross-assembly parallel bleed in the existing Core suite — out of this change's scope, but worth watching in CI.

- **NOVEL MECHANISM (first repo use of `#:project`):** `daemon/Daemon.cs` is an ADR-019 file-based program, but its first-party deps can't all be `#:package`: `Daemon.Routing` is `IsPackable=false`, and `Dmon.Tools.Dcal`/`Dmon.Tools.Dmail`/`Dmon.Memory` aren't packed by `scripts/pack-core.sh`. So `Daemon.cs` uses `#:project` directives for those four local projects + `#:package dmoncore@0.2.*` + the three provider packages (which ARE on the packed feed). No repo precedent for `#:project` — if it doesn't resolve in this SDK, the worker STOPS and reports 5.1 blocked (does not repackage the local projects). The 5.3 build (`dotnet build daemon/Daemon.cs -c Release`) is the proof.
- **Two orchestrator doc realignments applied (from architect findings):** (1) task 5.3 build command corrected `daemon.slnx` → `dotnet build daemon/Daemon.cs` (a file-based program is not a `.csproj`, not in any `.slnx`).
