# DEVLOG: conversation-persistence

Session-storage owns a lossless, dmon-owned **parts** conversation record; the memory tier derives its index from it; faithful session resume; write-time attachment offloading; un-quarantines `session.getMessages` (ADR-016).

## 1. Record and part types (additive)

- New `Dmon.Protocol.Conversation` namespace: abstract `SessionLogLine` (`[JsonPolymorphic("type")]`) → `MessageRecord`(`message`, `{entryId,timestamp,role,parts}`) + `CompactionMessage`(`compaction`); abstract `Part` (`[JsonPolymorphic("type")]`) → `text`/`toolCall`/`toolResult`/`image`/`reasoning`/`usage`/`unknown`. `args`/`result`/`raw` are `JsonElement` (BCL; intrinsic free-form, not a third-party type).
- **Prereq (orchestrator-sanctioned):** moved `CompactionMessage` from `Dmon.Core.Session` into the union (`: SessionLogLine`). Removed its manual `Type="compaction"` property; preserved the wire shape by changing `MessageAppender.AppendCompactionAsync` to serialize **through the base** (`Serialize<SessionLogLine>(compaction)`) so `"type":"compaction"` is still emitted. Namespace updates in `SessionStore` + 2 tests; compaction tests assert the on-wire `type` and stay green.
- **Note (read path):** `SessionStore` still deserializes a compaction line directly to the concrete `CompactionMessage` — works because STJ ignores the unmapped `type` by default. On record for when the read path is reworked (groups 3–5).
- **Decision:** added a `JsonSchemaExporter` test (`ConversationRecordSchemaTests`, 6 assertions) that exports schemas for `SessionLogLine` + `Part`, asserts success, and asserts the schema JSON contains **no** `Microsoft.Extensions.AI`/`ChatMessage`/`AIContent` — locking the "no third-party types in the API" principle as an automated guard rather than inspection. (`JsonSchemaExporter` on .NET 10 needs an explicit `DefaultJsonTypeInfoResolver` in the options.)
- **Review:** reviewer approved (no blockers; nits: the schema-export check — addressed via the new test; minimal `UsagePart` and doc-comment value sets for `role`/`reason` — accepted/deferred).
- Gates: `make build` 0/0; full `make test` green (Protocol 64, Core 559/+1 skip, Terminal 157, Gateway 123, …, 0 failures, 2 pre-existing skips); `openspec validate --strict` valid.

## 2. ChatMessage ⇆ record mapping

- New `ConversationMapper` (static, `Dmon.Core.Session`, `src/Dmon.Core/Session/ConversationMapper.cs`) — the single bidirectional bridge between M.E.AI `ChatMessage`/`AIContent` and the dmon-owned parts record. Placed in `Dmon.Core` (which references both M.E.AI and `Dmon.Protocol`); never in `Dmon.Protocol`, keeping the API definition third-party-free (ADR-016).
- **API:** `ToParts(ChatMessage, string producedBy="unknown") → (string Role, IReadOnlyList<Part>)` (write) and `ToMessage(MessageRecord) → ChatMessage` (replay).
- **Decision (entryId/timestamp ownership):** `ToParts` returns only `(role, parts)` and mints **no** `entryId`/`timestamp` — those belong to session-storage at append time (design D2, group 3). The mapper is content-only, so the round-trip test never depends on synthesised ids.
- **Write (D1):** `TextContent`→`TextPart`, `FunctionCallContent`→`ToolCallPart{callId,name,args}`, `FunctionResultContent`→`ToolResultPart{callId,result}` **inline** (`attachmentRef` null — offloading is group 3, D4); any other `AIContent`→`UnknownPart{raw,producedBy}`. Lenient: `MapUnknown` wraps its own `SerializeToElement` in try/catch with a type-name fallback so nothing throws or drops. `args`/`result` via `JsonSerializer.SerializeToElement` (default options, matching `MessageAppender`).
- **Replay subset (D1):** `ToMessage` reconstructs only `text`/`toolCall`/`toolResult`/`image`; `reasoning`/`usage`/`unknown` are preserved in the record but excluded from LLM context. `image` is best-effort (inline `dataBase64`→`DataContent`; `attachmentRef`-only skipped — file IO is groups 3/5).
- **Tests** (`test/Dmon.Core.Tests/Session/ConversationMapperTests.cs`, 6 facts): value-level round-trip for text/toolCall(args `"ls -la"`)/toolResult(`"file listing output"`); `UnknownPart` preserved on write + excluded on replay; reasoning/usage excluded on replay; serialized `MessageRecord` JSON asserted free of `Microsoft.Extensions.AI`/`ChatMessage`/`AIContent` and concrete content type names; structural confirmation `ToParts` exposes no id/timestamp.
- **Review:** reviewer approved with nits; addressed nit 1 (round-trip now asserts the actual arg/result *values*, not just presence) and nit 3 (removed the four `// ---` section-banner comments). Deferred nits 2 (`producedBy` default never falls back to type name — judgement call) and 4 (image/`result==null` replay branches uncovered — dormant forward-compat).
- Gates: `make build` 0/0; `make test` green (Core 565/+1 pre-existing skip, Protocol/Memory/Terminal/Gateway/BuiltinTools/Extensions/Runtime all pass, 0 failures); `openspec validate --strict` valid.

## 3. Session-storage canonical write path

- `SessionStore` gains two methods (both on `ISessionStore`): `AppendMessageAsync(sessionId, role, IReadOnlyList<Part> parts, ct) → entryId` and `ReadRecordsAsync(sessionId, applyCompaction=true, ct) → IReadOnlyList<SessionLogLine>`. Additive — not yet wired into the turn flow (group 4 calls `ConversationMapper.ToParts` then `AppendMessageAsync`).
- **Decision (entryId/timestamp minting):** `AppendMessageAsync` mints both `entryId` (`Guid`) and `timestamp` (`UtcNow`) and returns the `entryId` — session-storage owns the id space (design D2). This is exactly the minting group 4 strips out of `ShortTermMemory`.
- **Decision (discriminator):** the `message` record is serialized **through the `SessionLogLine` base** (`Serialize<SessionLogLine>(record)`) so `"type":"message"` is emitted, mirroring `MessageAppender.AppendCompactionAsync`. A bare `MessageRecord` serialize would drop the discriminator.
- **Decision (line write):** `SessionStore` writes the canonical line itself via a private `AppendLineAsync` helper rather than injecting `IMessageAppender` — keeps session-storage self-contained; `IMessageAppender` stays a `Dmon.Memory`-only registration (a premature `TryAddSingleton` in `AddDmonCore` was added then removed per review).
- **Write-time offloading (D4):** reuses the existing `IAttachmentStore` (injected into `SessionStore`), which owns the `Dmon:Session:AttachmentThresholdBytes` threshold read (default 1024). For each `ToolResultPart` with inline `Result`: derive full text (string `JsonElement` → `GetString()`, else `GetRawText()` — un-double-encoded), call `StoreIfLargeAsync`; on non-null path → `with { AttachmentRef = path, Result = <~200-char preview as JSON string>, Truncated = true }` (preserves `IsError`/`CallId`); on null → left inline. **Note (spec prose typo):** the spec says `Daemon:Session:AttachmentThresholdBytes`; the real key is `Dmon:Session:...` — reusing `IAttachmentStore` uses the correct key, no behaviour change.
- **Typed read-back:** `ReadRecordsAsync` deserializes each line via the polymorphic base (`Deserialize<SessionLogLine>`), **resolving the group-1 nit** (compaction no longer read as a concrete type). Applies the same position-based compaction rule as the existing `ReadMessagesAsync` (last marker by file position; skip records at-or-before the `supersedesUpTo` line — UUIDs aren't lexicographically monotonic). The legacy `ReadMessagesAsync(...)→IReadOnlyList<object>` is left intact (still used by `SessionHandler`/`ShortTermMemory` until later groups).
- **Tests:** new `test/Dmon.Core.Tests/Session/MessagePersistenceTests.cs` (11) — entryId minting/return, `type:"message"` discriminator, polymorphic round-trip, small-inline vs large-offloaded (attachment file holds full raw content), `IsError` preservation, non-`ToolResultPart` passthrough, compaction applied/not-applied, `CompactionMessage` polymorphic read. Constructor-signature fixes across `CompactionTests`/`SessionStoreTests`/`SessionForkCloneTests`/`SessionHandlerTypedEventsTests` (FakeSessionStore stub) + `Dmon.Memory.Tests` `TempSessionFixture`.
- **Review:** approved with nits; addressed both (dropped the dead `IMessageAppender` registration; simplified the test helper's tuple return). Reviewer also noted (deferred) a pre-existing path-traversal surface in `AttachmentStore` (`<callId>.txt` filename, no traversal guard) — harmless now (test-supplied callIds) but should be guarded before group 4 wires live tool-call ids.
- Gates: `make build` 0/0; `make test` green (Core 576/+11, all 10 projects pass, 2 pre-existing skips); `openspec validate --strict` valid.

## 4. Invert turn-completion wiring (cutover)

- **Pre-flight finding (the proposal's premise was wrong):** there was **no** turn→persistence wiring to "invert". `TurnHandler` never called memory; `IMemory` was registered but injected into nothing; `messages.jsonl` was never written in a live session (only by tests). `_history` holds only system/user/assistant-**text**; tool calls/results never enter it. `HandleToolCallAsync` (`TurnHandler.cs:467`) records a **placeholder** tool result — real results are gated on a separate "Group 9.5" `FunctionInvokingChatClient` integration. Surfaced to the architect; two decisions taken (below).
- **Architect decision 1 — chain `TurnHandler → ISessionStore → IMemory`.** `ISessionStore` is injected into `TurnHandler`; `IMemory` is injected (lazily) into `SessionStore`. **Session-storage is the orchestration point**: `AppendMessagesAsync` appends the canonical record(s) first, then drives the index. **Sanctioned deviation from design.md D2** (which named `TurnHandler` the orchestrator); the spec's "append before index" ordering still holds.
- **Architect decision 2 — persist real turns, skip stub tool results.** `TurnHandler` persists the new `_history` entries per turn (system excluded — the prompt is rebuilt on load per ADR-014), which is faithfully text-only today; no stub tool result is ever persisted. When Group 9.5 puts structured tool messages into `_history`, they flow through the same path unchanged.
- **DI cycle break:** `SessionStore → IMemory → ShortTermMemory → ISessionStore` is a construction cycle. Broken by registering `Lazy<IMemory?>` (`AddSingleton(sp => new Lazy<IMemory?>(() => sp.GetService<IMemory>()))`) and dereferencing `.Value` only inside `AppendMessagesAsync`. Nullable → canonical append always runs; indexing skipped when memory unresolved (and for the many direct `new SessionStore(...)` test constructions).
- **`SessionStore.AppendMessagesAsync(sessionId, IReadOnlyList<ChatMessage> messages, scope, ct) → entryIds`:** maps each message via `ConversationMapper.ToParts`, then a private `BuildAndWriteRecordAsync` helper mints `entryId`/`timestamp` **once**, runs offloading, builds + persists the `MessageRecord`, and **returns that exact record**. `AppendMessagesAsync` collects the written records and passes them verbatim to `IMemory.RecordAsync` — so the indexed record is reference-identical to disk (same entryId/timestamp/offloaded parts). `AppendMessageAsync` (group 3) now delegates to the helper and still returns `entryId` (public contract unchanged).
- **Decision (reviewer fix — ADR-016 fidelity):** the first cut rebuilt a fresh `MessageRecord` (second `UtcNow` + pre-offload parts) for memory, so disk↔index diverged on timestamp (and would diverge on tool-result content once 9.5 lands, breaking "index is a derivation of the canonical record"). Fixed by the shared-helper approach above; guarded by `AppendMessagesAsync_WithMemory_MemoryRecordMatchesDiskRecord` (asserts memory record == `ReadRecordsAsync` record).
- **Memory cutover:** `IMemoryStore.RecordAsync` (+ `Memory` facade + both tiers) signature `IReadOnlyList<ChatMessage>` → `IReadOnlyList<MessageRecord>` (index-only, keyed on supplied `entryId`; `MessageRecord` is dmon-owned, `Dmon.Abstractions` already references `Dmon.Protocol`). `ShortTermMemory` dropped `IMessageAppender` + the JSONL append; derives index text from `TextPart`s, keys on `record.EntryId` (no minting); `RebuildFromJsonlAsync`/`TryParseLineToEntry` parse `message` records via the polymorphic base and derive text from `TextPart`s. `TurnLineRecord` deleted.
- **Tests:** `Dmon.Core.Tests` — `TurnHandlerIntegrationTests` (`SpySessionStore`, `ActiveSessionHandler`, 3 persistence tests) + `SessionStoreTests` (canonical write, system-skip, memory-after-canonical, disk↔index parity). `Dmon.Memory.Tests` — `RecordAsync` call sites + fakes moved to `MessageRecord`; `RecordPathTests` rewritten (index-only, entryId keying, JSONL-untouched-by-RecordAsync); rebuild tests seed JSONL via `SessionStore.AppendMessageAsync`.
- Gates: `make build` 0/0; `make test` green (Core 583, Memory 51/+1 skip, all 10 projects pass, 2 pre-existing skips); `openspec validate --strict` valid; no `TurnLineRecord` refs remain.

## NEXT

- **Up next:** Group 5 — faithful session resume: on `session.load`, read the canonical record via `SessionStore.ReadRecordsAsync`, apply the compaction rule, and seed `TurnHandler._history` via the replay-subset mapping (`ConversationMapper.ToMessage`); after each completed turn is persisted, reconcile that turn's `_history` entries to the persisted preview form (D6) so subsequent live turns send shrunk historical tool results.
- **Open questions:** none (tool-result fidelity is a known Group 9.5 dependency, not a blocker for G5 — resume restores text/tool-calls; results follow when 9.5 lands).
- **Nits / deferred:** (1) **best-effort persistence, no retry** — `TurnHandler._persistedCount` advances in `finally`, so a failed `AppendMessagesAsync` logs but never retries → permanent loss of that turn from the session. Accepted for V1 (no production data); revisit for durability. (2) **rebuild defaults all entries to `MemoryScope.Agent`** (scope isn't carried on `MessageRecord`), so a `Session`-scoped live record becomes agent-scoped after rebuild — pre-existing; owner = whichever change persists memory scope. (3) **guard `AttachmentStore` against `callId` path-traversal before Group 9.5** wires live tool-call ids into the offload path (no `ToolResultPart`s are offloaded yet). (4) `ConversationMapper` `producedBy` default never reaches type-name fallback. (5) `ReadRecordsAsync`/`ReadMessagesAsync` duplicate the read+compaction scan — acceptable while `ReadMessagesAsync` is on its way out.
- **Carry-forward:** branch `change/conversation-persistence`. Apply gates: `dotnet clean -c Debug` before `make test` (stale-`bin/Debug` hazard).
